# JASCA 사용설명서

## 1. 문서 개요

JASCA는 Trivy 스캔 결과를 수집하고, 프로젝트/조직/정책 기준으로 취약점과 오픈소스 라이선스를 관리하는 보안 운영 플랫폼입니다. 이 문서는 실제 운영자가 각 화면 탭에서 무엇을 확인하고 어떤 순서로 사용하면 되는지 안내합니다.

기본 사용 흐름은 다음과 같습니다.

1. 프로젝트를 확인하거나 생성합니다.
2. Trivy JSON을 업로드하거나 검사 대상 파일/압축 파일을 직접 업로드해 스캔합니다.
3. 스캔 결과에서 취약점, 라이선스, 정책 위반 여부를 확인합니다.
4. 취약점 조치 상태를 관리하고 필요 시 예외를 요청합니다.
5. 관리자는 조직, 사용자, 정책, 수동 Advisory, Trivy DB, 알림 등을 운영 정책에 맞게 관리합니다.

## 2. 사용자 화면

### 2.1 대시보드

대시보드는 전체 보안 현황을 빠르게 보는 시작 화면입니다.

주요 기능:

- 전체 스캔 수, 취약점 수, 심각도별 분포를 확인합니다.
- 최근 스캔 결과와 프로젝트별 보안 상태를 확인합니다.
- Critical/High 취약점이 늘었는지 빠르게 파악합니다.
- AI 분석 버튼이 있는 경우 최근 변화 원인을 요약받을 수 있습니다.

사용 방법:

1. 로그인 후 좌측 메뉴에서 `대시보드`를 선택합니다.
2. 상단 통계 카드에서 전체 위험도를 확인합니다.
3. Critical/High가 많은 프로젝트를 우선 확인합니다.
4. 최근 스캔을 클릭해 상세 취약점으로 이동합니다.

운영 팁:

- 매일 최초 접속 시 대시보드에서 신규 Critical/High 증가 여부를 확인하는 것을 권장합니다.
- 스캔이 오래 갱신되지 않은 프로젝트는 최신 결과가 아닐 수 있으므로 재스캔 대상입니다.

### 2.2 프로젝트

프로젝트는 스캔 결과와 취약점을 묶는 기본 단위입니다.

주요 기능:

- 프로젝트 목록과 최신 스캔 상태를 확인합니다.
- 프로젝트별 취약점 현황, 라이선스 현황, 최근 스캔 이력을 봅니다.
- 프로젝트 상세에서 해당 프로젝트의 스캔 결과와 취약점만 필터링해 확인합니다.

사용 방법:

1. 좌측 메뉴에서 `프로젝트`를 선택합니다.
2. 프로젝트명을 클릭해 상세 화면으로 이동합니다.
3. 최신 스캔 일시와 취약점 수를 확인합니다.
4. 프로젝트별 라이선스 또는 취약점 상세로 이동해 조치합니다.

운영 팁:

- 프로젝트명은 업무 시스템명이나 서비스명과 맞추는 것이 좋습니다.
- CI/CD 또는 API 토큰으로 업로드할 때도 동일 프로젝트에 매핑되도록 이름을 통일하세요.

### 2.3 스캔 결과

스캔 결과는 Trivy 검사 결과가 저장되는 화면입니다.

주요 기능:

- 전체 스캔 목록을 조회합니다.
- 스캔 대상, 프로젝트, 상태, 취약점 수, 출처, 스캔 일시를 확인합니다.
- 스캔 결과를 검색/필터링/정렬합니다.
- 새 스캔 결과를 업로드하거나 검사 대상 파일을 직접 업로드합니다.
- 두 스캔을 비교해 취약점 변화량을 확인합니다.

대상 표시 방식:

- 목록의 `대상`에는 사용자가 업로드한 파일명 또는 이미지명이 표시됩니다.
- 상세 화면의 `검사 위치`에는 Trivy가 실제로 검사한 경로가 표시됩니다.
- 압축 파일을 업로드한 경우 목록에는 예: `source.tar.gz`, 상세에는 예: `/tmp/jasca-trivy-uploads/.../extracted`가 표시됩니다.

스캔 업로드 방법:

1. 좌측 메뉴에서 `스캔 결과`를 선택합니다.
2. `스캔 업로드`를 클릭합니다.
3. 업로드 방식 중 하나를 선택합니다.
4. 기존 Trivy JSON/SARIF 결과 파일을 업로드합니다.
5. 또는 검사 대상 파일/압축 파일을 업로드해 JASCA 서버에서 Trivy를 실행합니다.
6. 프로젝트를 선택하거나 프로젝트명을 입력합니다.
7. 저장 후 목록에서 결과를 확인합니다.

직접 Trivy 검사 지원 파일:

- 단일 파일
- `.zip`
- `.tar`
- `.tar.gz`
- `.tgz`

폐쇄망 Trivy 옵션:

- `offline scan`: 폐쇄망 환경에서 온라인 조회를 줄입니다.
- `skip DB update`: Trivy 취약점 DB 업데이트를 건너뜁니다.
- `skip Java DB update`: Java DB 업데이트를 건너뜁니다.
- `ignore unfixed`: 수정 버전이 없는 취약점을 제외합니다.
- `scanners`: `vuln`, `license`, `secret`, `config` 등 검사 종류를 선택합니다.
- `severity`: 표시할 심각도 범위를 제한합니다.
- `timeout`: 검사 제한 시간을 지정합니다.

검사 중지:

- 직접 Trivy 검사 중에는 `검사 중지` 버튼으로 진행 중인 검사를 취소할 수 있습니다.
- 취소한 검사 대상 파일은 정리 대상이며, 정상 완료된 업로드 파일도 검사 후 삭제됩니다.

상세 화면에서 확인할 내용:

- 대상: 업로드 파일명 또는 이미지명
- 검사 위치: Trivy 실제 검사 경로
- 프로젝트, 태그, 스캔 일시
- 취약점 목록
- 패키지 라이선스 정보
- 정책 평가 결과가 연결된 경우 정책 위반 여부

### 2.4 취약점

취약점 화면은 스캔에서 발견된 CVE와 수동 Advisory를 조치 상태별로 관리하는 곳입니다.

주요 기능:

- Critical/High/Medium/Low 심각도별 취약점을 조회합니다.
- 패키지명, CVE ID, 프로젝트, 상태별로 검색/필터링합니다.
- 취약점 상세에서 설명, 영향 패키지, 설치 버전, 수정 버전, 참고 링크를 확인합니다.
- 담당자 지정, 상태 변경, 댓글/이력 확인 기능을 사용할 수 있습니다.

사용 방법:

1. 좌측 메뉴에서 `취약점`을 선택합니다.
2. Critical/High 필터를 우선 적용합니다.
3. 취약점 상세에서 `패키지`, `현재 버전`, `수정 버전`을 확인합니다.
4. 담당자를 지정하고 조치 상태를 변경합니다.
5. 즉시 조치가 어려운 경우 정책/예외 절차를 사용합니다.

상태 관리 예시:

- `OPEN`: 새로 발견되었거나 아직 조치되지 않음
- `IN_PROGRESS`: 담당자가 조치 중
- `RESOLVED`: 조치 완료 확인
- `ACCEPTED_RISK`: 위험을 인지하고 예외 승인
- `FALSE_POSITIVE`: 오탐으로 판단

운영 팁:

- 동일 CVE라도 프로젝트별 패키지 버전과 위치가 다를 수 있으므로 상세에서 실제 영향 범위를 확인하세요.
- `[수동 Advisory]`로 시작하는 항목은 관리자가 직접 등록한 사내 취약점 룰에서 생성된 항목입니다.

### 2.5 라이선스

라이선스 화면은 보안 취약점이 아니라 오픈소스 라이선스 컴플라이언스를 관리하는 곳입니다.

주요 기능:

- 스캔 결과에 포함된 패키지 라이선스를 조회합니다.
- MIT, Apache-2.0, GPL-3.0, AGPL, SSPL 등 SPDX 라이선스를 확인합니다.
- 라이선스를 분류별로 확인합니다.
- 프로젝트별 라이선스 위험도를 확인합니다.

라이선스 분류 예시:

- `PERMISSIVE`: 일반적으로 사용이 쉬운 허용형 라이선스
- `NOTICE`: 고지 의무가 있는 라이선스
- `RECIPROCAL`: 약한 상호 공개 의무가 있을 수 있는 라이선스
- `RESTRICTED`: 강한 제약이 있어 검토가 필요한 라이선스
- `FORBIDDEN`: 조직 정책상 사용 금지 대상
- `UNKNOWN`: 자동 분류가 안 된 라이선스

사용 방법:

1. 좌측 메뉴에서 `라이선스`를 선택합니다.
2. `금지`, `제한`, `미확인` 라이선스를 우선 확인합니다.
3. 패키지명과 라이선스명을 확인합니다.
4. 필요 시 관리자에게 라이선스 정책 분류를 요청합니다.

주의사항:

- 라이선스 정보는 Trivy 결과에 license scanner 결과가 포함되어야 표시됩니다.
- 직접 검사 시 scanner 옵션에서 `license`를 포함해야 라이선스 정보가 충분히 수집됩니다.

### 2.6 Trivy DB

Trivy DB 화면은 취약점 DB 상태를 확인하고 폐쇄망 DB 반입/동기화 상태를 점검하는 화면입니다.

주요 기능:

- 현재 Trivy DB 정보와 메타데이터를 확인합니다.
- DB 업로드 또는 동기화 관련 기능을 사용합니다.
- CVE 또는 패키지 기준으로 DB 정보를 조회합니다.

사용 방법:

1. 좌측 메뉴에서 `Trivy DB`를 선택합니다.
2. DB 파일이 정상 인식되는지 확인합니다.
3. 폐쇄망에서 DB가 오래되었다면 외부망에서 최신 Trivy DB를 내려받아 반입합니다.
4. 반입 후 컨테이너에 Trivy cache 경로가 정상 마운트되었는지 확인합니다.

폐쇄망 운영 팁:

- 컨테이너는 호스트 Trivy 바이너리를 직접 쓰지 않고, 이미지 내부 Trivy CLI를 사용합니다.
- 대신 호스트에 있는 Trivy DB/cache 디렉터리를 `/app/trivy-db`로 마운트해 사용합니다.
- 신규 CVE 탐지는 Trivy DB 최신성에 영향을 받습니다.
- Trivy DB에 없는 긴급 취약점은 관리자 `수동 취약점`에서 Advisory로 등록해 보완할 수 있습니다.

### 2.7 정책

정책 화면은 스캔 결과에 대해 조직 보안 기준을 적용하는 곳입니다.

주요 기능:

- 심각도 기준 정책을 등록합니다.
- 특정 라이선스 사용 차단 정책을 등록합니다.
- 정책 평가 결과와 위반 여부를 확인합니다.
- 정책 예외 요청 흐름과 연결됩니다.

사용 방법:

1. 좌측 메뉴에서 `정책`을 선택합니다.
2. 새 정책을 생성합니다.
3. 적용 범위와 조건을 설정합니다.
4. 스캔 결과에 정책 평가를 실행하거나 자동 평가 결과를 확인합니다.
5. 정책 위반 항목은 취약점/예외 관리와 연계해 조치합니다.

정책 예시:

- Critical 취약점이 1건 이상이면 배포 불가
- High 취약점이 5건 이상이면 승인 필요
- AGPL, SSPL 라이선스 사용 금지
- 특정 프로젝트에만 완화된 기준 적용

### 2.8 API 토큰

API 토큰은 CI/CD나 외부 스캐너가 JASCA에 결과를 업로드할 때 사용합니다.

주요 기능:

- 사용자별 API 토큰을 생성합니다.
- 토큰 권한 범위를 설정합니다.
- CI/CD 스크립트에서 인증값으로 사용합니다.

사용 방법:

1. 좌측 메뉴에서 `API 토큰`을 선택합니다.
2. 새 토큰을 생성합니다.
3. 생성 직후 표시되는 토큰 값을 안전한 곳에 보관합니다.
4. CI/CD 환경변수에 등록합니다.
5. Trivy 결과 업로드 API 호출에 사용합니다.

주의사항:

- 토큰은 생성 직후에만 전체 값이 보일 수 있습니다.
- 퇴사자나 역할 변경 사용자의 토큰은 즉시 폐기하세요.

### 2.9 연동 가이드

연동 가이드는 Trivy 실행 명령과 JASCA 업로드 API 사용 예시를 제공하는 화면입니다.

주요 기능:

- 취약점 스캔 명령 예시 확인
- 라이선스 스캔 명령 예시 확인
- 폐쇄망 Trivy 옵션 확인
- CI/CD 연동 예시 확인
- API 업로드 방식 확인

사용 방법:

1. 좌측 메뉴에서 `연동 가이드`를 선택합니다.
2. 운영 환경에 맞는 탭을 선택합니다.
3. Trivy 명령 또는 curl 예시를 복사합니다.
4. 프로젝트 ID, API 토큰, 서버 URL을 실제 값으로 바꿔 사용합니다.

### 2.10 알림

알림 화면은 취약점, 정책 위반, 시스템 이벤트 등 사용자가 받아야 할 메시지를 확인하는 곳입니다.

주요 기능:

- 읽지 않은 알림 확인
- 알림 상세 확인
- 전체 읽음 처리

사용 방법:

1. 좌측 메뉴에서 `알림`을 선택합니다.
2. 새 알림을 확인합니다.
3. 조치가 필요한 항목은 관련 프로젝트/취약점/정책 화면으로 이동합니다.

### 2.11 설정

설정 화면은 개인 프로필, 알림 수신 설정, UI 옵션 등을 관리하는 곳입니다.

주요 기능:

- 사용자 프로필 확인
- 개인 알림 설정
- 화면/환경 설정
- SSO 또는 계정 관련 정보 확인

사용 방법:

1. 좌측 메뉴에서 `설정`을 선택합니다.
2. 필요한 개인 설정을 변경합니다.
3. 저장 후 다시 로그인하거나 새로고침해 반영 여부를 확인합니다.

### 2.12 사이트맵

사이트맵은 JASCA의 전체 화면 구조를 빠르게 찾는 탐색 화면입니다.

주요 기능:

- 사용자 메뉴와 관리자 메뉴의 주요 화면을 한눈에 확인합니다.
- 원하는 기능 화면으로 빠르게 이동합니다.

## 3. 관리자 화면

관리자 화면은 우측 상단 `관리자` 버튼 또는 `/admin` 경로로 접근합니다. 접근 가능 역할은 주로 `SYSTEM_ADMIN`, `ORG_ADMIN`, `SECURITY_ADMIN`입니다.

### 3.1 관리자 대시보드

전체 조직 기준의 보안 운영 현황을 확인합니다.

주요 기능:

- 전체 프로젝트, 사용자, 스캔, 취약점 현황 확인
- 심각도별 취약점 분포 확인
- 최근 위험 변화 확인

사용 방법:

1. 관리자 화면에서 `대시보드`를 선택합니다.
2. 전체 위험도를 확인합니다.
3. 취약점 현황 또는 프로젝트 관리로 이동해 상세 조치합니다.

### 3.2 조직 관리

조직은 사용자, 프로젝트, 정책 범위를 나누는 기준입니다.

주요 기능:

- 조직 생성/수정/삭제
- 조직별 프로젝트와 사용자 구분
- 조직 단위 접근 권한 관리

사용 방법:

1. `조직 관리`를 선택합니다.
2. 필요한 조직을 생성합니다.
3. 사용자와 프로젝트를 해당 조직에 연결합니다.

운영 팁:

- 실제 회사 조직, 서비스 조직, 보안 관리 단위에 맞게 너무 잘게 쪼개지 않도록 설계하세요.

### 3.3 사용자 관리

사용자 계정과 역할을 관리합니다.

주요 기능:

- 사용자 생성/수정/비활성화
- 역할 부여
- 조직/프로젝트 범위 권한 설정

주요 역할:

- `SYSTEM_ADMIN`: 전체 시스템 관리자
- `ORG_ADMIN`: 조직 관리자
- `SECURITY_ADMIN`: 보안 운영 관리자
- `PROJECT_ADMIN`: 프로젝트 관리자
- `DEVELOPER`: 프로젝트 취약점 확인/조치 담당
- `VIEWER`: 조회 전용 사용자

사용 방법:

1. `사용자 관리`를 선택합니다.
2. 사용자를 생성하거나 기존 사용자를 선택합니다.
3. 조직과 역할을 지정합니다.
4. 필요 시 프로젝트 범위 권한을 제한합니다.

주의사항:

- 관리자 권한은 최소 인원에게만 부여하세요.
- 사용자가 조직에 속하지 않으면 일부 프로젝트/스캔이 보이지 않을 수 있습니다.

### 3.4 프로젝트 관리

사용 방법:

3. 조직, 설명, 담당 범위를 정리합니다.
4. 스캔 결과가 올바른 프로젝트로 들어오는지 확인합니다.

### 3.5 취약점 현황

3. 오래 미조치된 취약점이나 반복 발생 취약점을 우선 점검합니다.

### 3.6 수동 취약점

수동 취약점 Advisory는 Trivy DB에 아직 없거나 사내에서 긴급하게 관리해야 하는 취약점을 관리자가 직접 등록하는 기능입니다.

주요 기능:

- Advisory ID 등록
- CVE ID 등록 또는 미지정
- 제목, 설명, 심각도 설정
- 패키지명과 영향 버전 범위 지정
- 수정 버전과 조치 가이드 입력
- 참고 URL 등록
- 전체/조직/프로젝트 범위 지정
- 활성/비활성 전환

사용 방법:

1. 관리자 메뉴에서 `수동 취약점`을 선택합니다.
2. `Advisory 추가`를 클릭합니다.
3. 패키지명과 영향 버전 범위를 입력합니다.
4. 심각도, 수정 버전, 조치 가이드를 입력합니다.
5. 적용 범위를 선택합니다.
6. 저장합니다.
7. 이후 새 스캔부터 해당 Advisory가 패키지/버전 기준으로 매칭되어 취약점으로 표시됩니다.

대량 등록:

- `CSV/JSON 가져오기` 버튼으로 여러 Advisory를 한 번에 반입할 수 있습니다.
- 동일한 `advisoryId`가 이미 있으면 새로 만들지 않고 기존 항목을 수정합니다.
- CSV 헤더 예시는 다음과 같습니다.

```csv
advisoryId,cveId,title,description,severity,packageName,affectedVersionRange,fixedVersion,remediation,references,isActive,organizationId,projectId
KCB-ADV-2026-001,,긴급 사내 취약점,내부 공지 기준,HIGH,example-package,<1.2.3,1.2.3,패키지 업그레이드 필요,https://intra.example/advisory,true,,
```

API 자동화:

- 단건 등록: `POST /api/manual-advisories`
- 대량 등록: `POST /api/manual-advisories/bulk`
- 파일 반입: `POST /api/manual-advisories/upload` multipart field name은 `file`입니다.
- 자동화 계정은 `SYSTEM_ADMIN`, `ORG_ADMIN`, `SECURITY_ADMIN` 중 하나의 권한이 필요합니다.

버전 범위 예시:

- `*`: 모든 버전
- `<2.0.0`: 2.0.0 미만
- `<=1.5.0`: 1.5.0 이하
- `>=1.0.0 <1.2.3`: 1.0.0 이상 1.2.3 미만
- `1.0.0-1.4.0`: 1.0.0 이상 1.4.0 이하

주의사항:

- 저장 후 기존 스캔에는 자동 반영되지 않습니다. 기존 결과에 반영하려면 재스캔해야 합니다.
- Trivy DB 업데이트를 대체하는 기능이 아니라, 폐쇄망/긴급 상황에서 보완하기 위한 기능입니다.

### 3.7 라이선스 관리

조직 전체의 오픈소스 라이선스 현황과 분류 기준을 관리합니다.

주요 기능:

- 기본 라이선스 카탈로그 등록
- SPDX ID별 분류 확인
- 금지/제한/미확인 라이선스 확인
- 프로젝트별 라이선스 현황 확인

사용 방법:

1. `라이선스 관리`를 선택합니다.
2. 최초 운영 시 `기본 라이선스 등록`을 실행합니다.
3. `FORBIDDEN`, `RESTRICTED`, `UNKNOWN` 항목을 우선 검토합니다.
4. 조직 정책에 맞게 라이선스 분류 기준을 정리합니다.

### 3.8 정책 관리

조직 보안 정책과 평가 기준을 관리합니다.

주요 기능:

- 정책 생성/수정/삭제
- 정책 규칙 추가

- 정책 평가 실행

사용 방법:

1. `정책 관리`를 선택합니다.
2. 새 정책을 생성합니다.
3. 정책 조건과 적용 범위를 설정합니다.
4. 저장 후 스캔 결과에서 정책 위반 여부를 확인합니다.

운영 팁:

- 처음부터 너무 강한 정책을 적용하면 예외가 폭증할 수 있습니다.
- Critical/High 기준부터 단계적으로 적용하는 것이 좋습니다.

예외 승인은 특정 취약점 또는 정책 위반을 일정 기간 허용할 때 사용합니다.

주요 기능:

3. 승인 시 만료일을 명확히 설정합니다.
4. 반려 시 보완 사유를 남깁니다.

### 3.17 AI 설정, AI 프롬프트, AI 사용 기록

AI 관련 메뉴는 취약점 분석과 변화 분석에 사용하는 AI 기능을 설정하고 이력을 확인하는 영역입니다.

AI 설정:

- AI Provider, 모델, API URL, 키 등을 설정합니다.
- 폐쇄망에서는 내부 LLM 또는 vLLM/Ollama 연동 여부를 확인합니다.
- 사내에서 OpenAI 호환 API(`/v1/chat/completions`)로 모델을 서빙하는 경우 `vLLM` 또는 `Custom`을 선택하고 API URL과 모델명을 입력합니다.
- 모델 목록 조회(`/v1/models`)가 막혀 있거나 빈 목록을 반환하는 서버는 `모델명 직접 입력`을 켜고 운영자가 받은 정확한 모델명을 입력합니다.
- `Custom` 제공자는 사내 OpenAI 호환 서버 호출용이며, API Key가 필요한 경우에만 키를 입력합니다.

AI 프롬프트:

- 취약점 설명, 스캔 변화 분석 등에 사용하는 프롬프트를 관리합니다.
- 조직의 보고 문체나 분석 기준에 맞게 수정할 수 있습니다.

AI 사용 기록:

- AI 요청 이력, 사용량, 실패 여부를 확인합니다.
- 과도한 사용이나 오류를 점검합니다.

## 4. 폐쇄망 운영 절차

### 4.1 배포 전 백업

기존 운영 서버가 `/app/jasca/pgdata`, `/app/jasca/redis`를 사용한다면 배포 전 다음과 같이 백업합니다.

```bash
cd /app/jasca
tar -czf pgdata-backup-before-jasca-$(date +%Y%m%d-%H%M%S).tar.gz pgdata
tar -czf redis-backup-before-jasca-$(date +%Y%m%d-%H%M%S).tar.gz redis
```

### 4.2 신규 버전 반입

GitHub Release에서 오프라인 번들을 받아 폐쇄망으로 반입합니다.

```bash
tar -xzf jasca-offline-scan-target-display-20260622-bundle.tar.gz
cd jasca-offline-scan-target-display-20260622
cp deploy-existing-layout.env.example deploy-existing-layout.env
vi deploy-existing-layout.env
chmod +x deploy-existing-layout.sh
./deploy-existing-layout.sh
docker logs -f jasca
```

운영 서버 기준 예시:

```bash
APP_DIR=/app/jasca
IMAGE_NAME=jasca-offline:latest
CONTAINER_NAME=jasca
WEB_PORT=3005
CORS_ORIGIN=https://jasca.koreacb.com
JWT_SECRET=jasca_offline_secret
DB_PASSWORD=jasca_secret
DATABASE_URL=postgresql://jasca:${DB_PASSWORD}@localhost:5432/jasca
REDIS_URL=redis://localhost:6379
TRIVY_CACHE_MOUNT=/root/.cache/trivy
HOSTS_MOUNT=/etc/hosts
```

### 4.3 롤백

문제가 발생하면 새 컨테이너를 중지하고 기존 이미지로 다시 실행합니다. 데이터까지 되돌려야 할 경우 배포 전 백업한 `pgdata`, `redis`를 복원합니다.

```bash
docker stop jasca
docker rm jasca
```

이후 기존 배포 스크립트 또는 기존 이미지 실행 명령을 사용합니다.

## 5. 운영 체크리스트

일일 점검:

- 대시보드에서 신규 Critical/High 확인
- 최근 스캔 실패 여부 확인
- 오래된 스캔 프로젝트 확인
- 신규 예외 요청 확인

주간 점검:

- Trivy DB 최신성 확인
- 금지/제한 라이선스 확인
- 미조치 Critical/High 취약점 리뷰
- API 토큰과 사용자 권한 점검

월간 점검:

- 조직/프로젝트 구조 정리
- 퇴사자/이동자 권한 회수
- 정책 기준 재검토
- 수동 Advisory 만료 또는 불필요 항목 정리
- 백업과 롤백 절차 점검
